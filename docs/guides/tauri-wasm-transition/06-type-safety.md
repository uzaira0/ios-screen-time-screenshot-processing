# Chapter 06: Type Safety

## 1. The Contract Problem

Every client-server application has a contract boundary: the JSON that crosses the wire between backend and frontend. In a fast-moving codebase, this boundary is where bugs hide.

The failure mode is always the same:

1. A backend developer adds a field to a Pydantic model or renames an enum variant.
2. The frontend continues compiling because TypeScript has no way to know the shape changed.
3. The app ships. Users hit a runtime crash -- `Cannot read properties of undefined (reading 'username')` -- because the field moved from `username` to `user_name` three PRs ago.

This is **contract drift**. It is the single most common source of production bugs in full-stack TypeScript/Python projects that rely on manual type definitions. The root cause is always the same: the frontend defines its own copy of a type that mirrors the backend, and nobody keeps them synchronized.

The problem compounds in a dual-mode system. This project runs in three modes -- server, WASM, and Tauri. The server mode talks to FastAPI over HTTP. The WASM and Tauri modes use IndexedDB locally but must produce data structures compatible with the server for eventual sync. If the contract drifts, sync silently corrupts data.

The solution is mechanical: **generate frontend types from backend schemas, automatically, on every change, and fail the build if they diverge.**

## 2. Pydantic as Single Source of Truth

Pydantic models in `src/screenshot_processor/web/database/schemas.py` are the canonical definition of every data structure that crosses any boundary in the system. This is non-negotiable.

```python
# src/screenshot_processor/web/database/schemas.py

class Point(BaseModel):
    """A 2D point with x, y coordinates (pixel positions)."""
    x: int = Field(..., ge=0)
    y: int = Field(..., ge=0)

class AnnotationCreate(BaseModel):
    screenshot_id: int
    hourly_values: dict[str, float]
    extracted_title: str | None = None
    extracted_total: str | None = None
    grid_upper_left: Point | None = None
    grid_lower_right: Point | None = None
    notes: str | None = None
```

FastAPI automatically generates an OpenAPI 3.1 specification from these Pydantic models. Every field name, every type constraint, every `Optional` annotation, every enum variant -- all of it appears in the spec. The spec is the machine-readable contract.

Key design rules:

- **All API request and response bodies must use Pydantic models.** No raw dicts.
- **Enums are defined once in `models.py` and imported into `schemas.py`.** The `ProcessingStatus`, `AnnotationStatus`, and `UserRole` enums exist in one place.
- **Field validators live on the Pydantic model.** If a field has constraints (e.g., `ge=0` on a pixel coordinate), the constraint is enforced by Pydantic and documented in the OpenAPI spec.
- **Computed fields use `@computed_field`.** They appear in the schema and the generated types.

When you change a Pydantic model, you change the contract. The OpenAPI spec changes. The generated TypeScript types change. The frontend compiler catches the mismatch. That is the entire point.

## 3. OpenAPI-to-TypeScript Generation

The generation pipeline has three steps:

1. **Extract the OpenAPI spec** from the running FastAPI application.
2. **Transform it to TypeScript** using `openapi-typescript`.
3. **Write the output** to `frontend/src/types/api-schema.ts`.

The `generate:api-types` script in `frontend/package.json` does all three:

```json
{
  "scripts": {
    "generate:api-types": "cd .. && python -c \"import sys; sys.path.insert(0, 'src'); from screenshot_processor.web.api.main import app; import json; print(json.dumps(app.openapi(), indent=2))\" > frontend/openapi.json && cd frontend && bunx openapi-typescript ./openapi.json -o src/types/api-schema.ts"
  }
}
```

Breaking this down:

1. `python -c "..."` imports the FastAPI `app` object and calls `app.openapi()` to get the spec as a Python dict, then serializes it to JSON. This does not start the server -- it just loads the app module and extracts the schema.
2. The JSON is written to `frontend/openapi.json` (an intermediate artifact, not checked in).
3. `bunx openapi-typescript ./openapi.json -o src/types/api-schema.ts` reads the spec and generates a single TypeScript file with `paths`, `components`, and `operations` type exports.

The generated file begins with:

```typescript
/**
 * This file was auto-generated by openapi-typescript.
 * Do not make direct changes to the file.
 */

export interface paths {
    "/": { /* ... */ };
    "/health": { /* ... */ };
    "/api/v1/auth/status": { /* ... */ };
    // ... every route
}
```

**When to regenerate:**

- After any change to a Pydantic schema in `schemas.py`.
- After adding, removing, or modifying an API route.
- After changing a route's query parameters, path parameters, or response model.
- After changing an enum in `models.py` that is used in a schema.

**How to run it:**

```bash
cd frontend && bun run generate:api-types
```

The backend Python environment must be available (the script imports the FastAPI app). In Docker-based development, run this from the backend container or ensure the Python packages are installed on the host.

## 4. Typed API Client

The generated types are consumed by `openapi-fetch`, a zero-overhead typed HTTP client. The client is configured in `frontend/src/services/apiClient.ts`:

```typescript
import createClient from "openapi-fetch";
import type { paths, components } from "@/types/api-schema";

const apiClient = createClient<paths>({ baseUrl: API_BASE_URL });
```

The `createClient<paths>()` call produces a client where every `.GET()`, `.POST()`, `.PUT()`, `.DELETE()`, and `.PATCH()` call is type-checked against the OpenAPI spec. The path string is a union of known paths. The `params` and `body` are inferred from the spec. The response type is inferred from the spec.

```typescript
// Fully typed -- path, params, body, and response are all inferred
const { data, error } = await apiClient.GET("/api/v1/screenshots/{screenshot_id}", {
  params: { path: { screenshot_id: 42 } },
});
// data is typed as ScreenshotRead | undefined
// error is typed as the error response schema
```

The `api` wrapper object provides a cleaner interface for application code:

```typescript
export const api = {
  screenshots: {
    async getById(id: number) {
      const { data, error } = await apiClient.GET(
        "/api/v1/screenshots/{screenshot_id}",
        { params: { path: { screenshot_id: id } } },
      );
      throwIfError(error, "Failed to get screenshot");
      return data;
    },
  },
};
```

**Re-exporting types for convenience:**

```typescript
// In apiClient.ts -- re-export commonly used types
export type GroupVerificationSummary = components["schemas"]["GroupVerificationSummary"];
export type ScreenshotTierItem = components["schemas"]["ScreenshotTierItem"];
export type ScreenshotComparison = components["schemas"]["ScreenshotComparison"];
```

Components then import from the wrapper:

```typescript
import { api } from "@/services/apiClient";
import type { GroupVerificationSummary } from "@/services/apiClient";
```

### Anti-patterns

Every one of these has caused a production bug in this project or a project like it:

```typescript
// WRONG -- Defining types manually in the frontend
// This WILL drift from the backend. It always does.
interface UserActivity {
  id: number;
  username: string;
  email: string;  // Backend renamed this to email_address two weeks ago
}

// WRONG -- Using raw fetch() with no type safety
const response = await fetch(`${API_URL}/admin/users`);
const data = await response.json();  // data is `any` -- invisible at compile time

// WRONG -- Creating duplicate axios/fetch instances
const api = axios.create({ baseURL: API_URL });
const { data } = await api.get("/users");  // data is `any`
```

```typescript
// CORRECT -- Import types from the generated OpenAPI schema
import type { components } from "@/types/api-schema";
type UserActivity = components["schemas"]["UserStatsRead"];

// CORRECT -- Use the typed apiClient
import { api } from "@/services/apiClient";
const users = await api.admin.getUsers();  // Return type inferred from schema

// CORRECT -- For one-off type references
type AnnotationCreate = components["schemas"]["AnnotationCreate"];
```

**The rule is simple: if you find yourself writing a TypeScript interface that mirrors a backend model, stop. Use the generated type.**

## 5. Contract Drift CI Check

Generation alone is not enough. Developers forget to regenerate. The CI pipeline must enforce it.

Add a job that regenerates the types and checks for uncommitted changes:

```yaml
# .github/workflows/ci.yml
contract-drift:
  name: Contract Drift Check
  runs-on: ubuntu-latest
  needs: [backend-install]
  steps:
    - uses: actions/checkout@v4

    - name: Set up Python
      uses: actions/setup-python@v5
      with:
        python-version: "3.11"

    - name: Install backend dependencies
      run: pip install -e ".[web]"

    - name: Set up Bun
      uses: oven-sh/setup-bun@v2

    - name: Install frontend dependencies
      run: cd frontend && bun install --frozen-lockfile

    - name: Regenerate API types
      run: cd frontend && bun run generate:api-types

    - name: Check for drift
      run: |
        git diff --exit-code frontend/src/types/api-schema.ts || {
          echo "::error::API types are stale. Run 'cd frontend && bun run generate:api-types' and commit the result."
          git diff frontend/src/types/api-schema.ts
          exit 1
        }
```

This job:

1. Installs the backend Python environment (needed to import the FastAPI app).
2. Installs the frontend Node/Bun environment (needed for `openapi-typescript`).
3. Regenerates `api-schema.ts` from the current backend code.
4. Runs `git diff --exit-code` on the generated file. If the file changed, the types were stale and the PR fails.

The error message tells the developer exactly what to do. No ambiguity.

**Cost:** ~30 seconds of CI time. **Value:** eliminates an entire class of production bugs permanently.

## 6. basedpyright

This project uses [basedpyright](https://github.com/DetachHead/basedpyright) for Python type checking instead of mypy. The configuration lives in `pyproject.toml`:

```toml
[tool.basedpyright]
include = ["src", "tests"]
exclude = [
    "**/__pycache__",
    "**/node_modules",
    "**/.venv",
    "**/venv",
    "frontend",
    "alembic/versions",
]
typeCheckingMode = "standard"
pythonVersion = "3.11"
pythonPlatform = "All"
reportMissingImports = true
reportMissingTypeStubs = false
reportUnusedImport = "warning"
reportUnusedVariable = "warning"
reportDuplicateImport = "warning"
```

### Why basedpyright over mypy

| Dimension | basedpyright | mypy |
|-----------|-------------|------|
| **Speed** | Incremental analysis, sub-second on warm cache | Full-project analysis, often 10-30s |
| **Pydantic support** | Native understanding of `BaseModel`, `Field`, `computed_field`, validators | Requires `pydantic-mypy` plugin, still has gaps |
| **Strictness granularity** | Per-rule severity (`error`, `warning`, `false`, `true`) | Mostly binary (on/off) per flag |
| **Editor integration** | Based on Pyright, same engine as Pylance in VS Code | Separate daemon, can lag |
| **Enum/Literal handling** | Narrowing works correctly with `StrEnum` and `Literal` unions | Known issues with `StrEnum` narrowing |
| **SQLAlchemy** | Better inference for mapped columns, relationship types | Requires `sqlalchemy-stubs` or `sqlalchemy[mypy]` plugin |

For a FastAPI + Pydantic + SQLAlchemy codebase, basedpyright catches more real bugs with fewer false positives. The `reportAttributeAccessIssue = "warning"` setting is important for SQLAlchemy ORM patterns where dynamic attribute access is legitimate.

The `dev` extra in `pyproject.toml` includes basedpyright:

```toml
dev = [
    "basedpyright>=1.18.0",
    # ...
]
```

Run it:

```bash
basedpyright src/ tests/
```

## 7. TypeScript Strict Mode

The frontend `tsconfig.json` enables every strictness flag that catches real bugs. Here is the full linting section with an explanation of each flag:

```json
{
  "compilerOptions": {
    "strict": true,
    "exactOptionalPropertyTypes": true,
    "noUnusedLocals": true,
    "noUnusedParameters": true,
    "noFallthroughCasesInSwitch": true,
    "noUncheckedIndexedAccess": true,
    "noImplicitOverride": true,
    "forceConsistentCasingInFileNames": true,
    "erasableSyntaxOnly": true
  }
}
```

### `strict: true`

This is a meta-flag that enables seven sub-flags at once: `strictNullChecks`, `strictFunctionTypes`, `strictBindCallApply`, `strictPropertyInitialization`, `noImplicitAny`, `noImplicitThis`, and `alwaysStrict`. The most important is `strictNullChecks`.

**Bug class prevented -- null dereferencing:**

```typescript
// Without strictNullChecks:
function getUser(id: number): User | undefined { /* ... */ }
const user = getUser(42);
console.log(user.name);  // Compiles fine. Crashes at runtime if user is undefined.

// With strictNullChecks:
const user = getUser(42);
console.log(user.name);  // ERROR: 'user' is possibly 'undefined'.
// Fix:
if (user) {
  console.log(user.name);  // OK
}
```

### `exactOptionalPropertyTypes: true`

Distinguishes between "property is missing" and "property is explicitly `undefined`".

**Bug class prevented -- accidental undefined assignment:**

```typescript
interface Config {
  timeout?: number;  // means: property may be absent
}

// Without exactOptionalPropertyTypes:
const config: Config = { timeout: undefined };  // Allowed. But did the developer mean "no timeout" or "forgot to set it"?

// With exactOptionalPropertyTypes:
const config: Config = { timeout: undefined };  // ERROR: 'undefined' is not assignable to 'number'.
// If you truly want undefined, declare: timeout?: number | undefined;
```

This matters for API payloads. Sending `{ "timeout": null }` is semantically different from omitting the field entirely. Pydantic treats them differently.

### `noUnusedLocals: true`

**Bug class prevented -- dead code masking real intent:**

```typescript
// ERROR: 'result' is declared but its value is never read.
const result = await api.screenshots.getById(42);
// Developer renamed the variable below but forgot to update the assignment.
return screenshotData;  // Where did screenshotData come from? Probably a bug.
```

### `noUnusedParameters: true`

**Bug class prevented -- API misunderstanding:**

```typescript
// ERROR: 'groupId' is declared but its value is never read.
async function fetchScreenshots(groupId: string, page: number) {
  // Developer forgot to pass groupId to the API call.
  return api.screenshots.list({ page });
}
```

Prefix with underscore to explicitly mark intentionally unused parameters: `_groupId`.

### `noFallthroughCasesInSwitch: true`

**Bug class prevented -- missing break in switch:**

```typescript
// ERROR: Fallthrough case in switch.
switch (status) {
  case "pending":
    showSpinner();
    // Developer forgot `break;` -- falls through to "completed"
  case "completed":
    showResult();
    break;
}
```

### `noUncheckedIndexedAccess: true`

**Bug class prevented -- array/object out-of-bounds access:**

```typescript
const hours = ["12AM", "1AM", "2AM"];
const label = hours[25];
// Without noUncheckedIndexedAccess: label is string
// With noUncheckedIndexedAccess: label is string | undefined

// Forces you to handle the case:
const label = hours[25];
if (label !== undefined) {
  renderLabel(label);
}
```

This is critical for the hourly data arrays in this project. Screenshot processing produces 24-element arrays indexed by hour. Off-by-one errors silently return `undefined` without this flag.

### `noImplicitOverride: true`

**Bug class prevented -- accidental method shadowing:**

```typescript
class WASMScreenshotService extends BaseScreenshotService {
  // ERROR: This member must have an 'override' modifier because it overrides a member in the base class.
  async getById(id: number) { /* ... */ }

  // CORRECT:
  override async getById(id: number) { /* ... */ }
}
```

When the base class renames a method, the subclass method silently becomes dead code without `override`. With this flag, the compiler tells you the override relationship broke.

### `forceConsistentCasingInFileNames: true`

**Bug class prevented -- case-sensitivity differences between macOS and Linux:**

```typescript
// On macOS (case-insensitive filesystem):
import { api } from "@/services/ApiClient";  // Works on Mac, fails on Linux CI

// With forceConsistentCasingInFileNames:
// ERROR: File name 'ApiClient.ts' differs from already included file name 'apiClient.ts' only in casing.
```

### `erasableSyntaxOnly: true`

Ensures that all TypeScript-specific syntax (type annotations, interfaces, type aliases) can be removed by simply erasing it, without requiring any transformation. This is important for compatibility with tools like Bun and esbuild that strip types without running the full TypeScript compiler.

**Bug class prevented -- runtime-dependent TypeScript features:**

```typescript
// ERROR with erasableSyntaxOnly:
enum Direction { Up, Down }  // Enums generate runtime code; use `const enum` or string unions

// CORRECT:
type Direction = "up" | "down";
```

## 8. Shared Types Between Modes

The `@/types/` directory contains type definitions that are mode-agnostic. Both server and WASM/Tauri implementations import from it:

```
frontend/src/types/
  api-schema.ts      # Generated from OpenAPI -- server contract types
  index.ts           # Re-exports of shared application types (Screenshot, Annotation, etc.)
```

The service interfaces in `@/core/interfaces/` define contracts using these shared types:

```typescript
// @/core/interfaces/IScreenshotService.ts
import type { Screenshot } from "@/types";

export interface IScreenshotService {
  getById(id: number): Promise<Screenshot>;
  list(params: ListParams): Promise<PaginatedResult<Screenshot>>;
  // ...
}
```

Both `APIScreenshotService` (server mode) and `WASMScreenshotService` (local mode) implement this interface. The `Screenshot` type is the same in both cases. The difference is the storage and retrieval mechanism, not the data shape.

Implementation-specific types stay inside the implementation:

```
frontend/src/core/implementations/
  server/
    APIScreenshotService.ts     # Uses openapi-fetch, types from api-schema.ts
  wasm/
    WASMScreenshotService.ts    # Uses Dexie, types from @/types
    storage/
      database/ScreenshotDB.ts  # Dexie-specific types (ImageBlob, QueueItem, SyncRecord)
    processing/
      workers/types.ts          # Worker message protocol types (internal to WASM)
    sync/
      SyncService.ts            # Sync-specific types (SyncConfig, HealthCheckResult)
```

The worker message types (`WorkerMessage`, `WorkerResponse`, `ProcessImagePayload`) are internal to the WASM processing pipeline. They never cross the service boundary. Components interact with `IProcessingService`, not with workers directly.

The Dexie schema types (`ImageBlob`, `QueueItem`, `SyncRecord`, `SyncStatus`) are internal to the WASM storage layer. They model IndexedDB tables, not API contracts. The `SyncRecord` type exists to track the relationship between local IDs and server IDs during sync -- it has no backend equivalent.

**The boundary rule:** if a type appears in a service interface (`I*.ts`), it belongs in `@/types/`. If it is specific to how one mode implements that interface, it belongs next to the implementation.
